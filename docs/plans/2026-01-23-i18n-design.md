# UI i18n 多语言设计（默认中文，可切换英文）

目标：在 Lightdash 前端引入 i18n 支持，默认语言为简体中文（zh-CN），支持英文切换（en），并将用户语言偏好保存在后端，保证跨设备一致。

## 1) 架构

- 采用 `react-i18next` 作为前端 i18n 方案，与 `packages/sdk-test-app` 保持一致技术栈。
- 翻译资源以本地静态 JSON 形式存放在 `packages/frontend/src/i18n/locales/{zh-CN,en}/`，按模块拆分 namespace（如 `common/auth/dashboard/explore/settings`）。
- 应用入口（`packages/frontend/src/index.tsx` 或 `App.tsx`）包裹 `I18nextProvider`，全局可用 `useTranslation`。
- 语言选择优先级：用户偏好（后端保存） > 浏览器语言（LanguageDetector） > `zh-CN`。
- UI 入口放在右上角用户菜单（`UserMenu`）中，首期不新增设置页入口，后续可扩展。

## 2) 数据流

- 启动时初始化 i18n（`fallbackLng: 'zh-CN'`，`supportedLngs: ['zh-CN', 'en']`），启用浏览器语言检测。
- `AppProvider` 拉取 `user` 后，如果存在 `preferredLanguage`，调用 `i18n.changeLanguage(preferredLanguage)` 覆盖初始语言；否则保持浏览器语言，最终回退 `zh-CN`。
- 语言切换流程：
  1) 用户菜单点击语言项
  2) 前端立即 `changeLanguage`
  3) `PATCH /user/me` 保存 `preferredLanguage`
  4) 刷新 `user` query（已有逻辑）
- 未登录页面仅使用浏览器语言检测 + `zh-CN` 回退，不写入偏好。

## 3) 组件与界面

- 新增 `LanguageSwitcher` 组件，放在 `UserMenu` 下拉菜单内。
- 菜单结构示例：
  - `Menu.Label`：Language / 语言
  - `Menu.Item`：简体中文（zh-CN）
  - `Menu.Item`：English（en）
- 当前语言显示选中态（图标或右侧勾选）。
- 首期覆盖核心路径文案：登录/首页/探索/仪表盘/图表/设置；非核心页面后续迭代补齐。

## 4) 后端与类型变更

- 在 `users` 表增加 `preferred_language` 字段（nullable，字符串），并通过迁移创建。
- `LightdashUser` 与 `UpdateUserArgs` 增加 `preferredLanguage?: string`。
- `UserModel.updateUser` 支持更新 `preferredLanguage`；`mapDbUserDetailsToLightdashUser` 返回该字段。
- `PATCH /user/me` 支持接收并保存 `preferredLanguage`。

## 5) 错误处理与边界

- 将 `supportedLngs` 限定为 `['zh-CN', 'en']`，并将浏览器语言 `zh`/`zh-Hans` 映射为 `zh-CN`。
- 后端返回的 `preferredLanguage` 不在支持列表时忽略，并回退 `zh-CN`。
- 翻译缺失时使用 `t('key', 'English fallback')` 兜底，避免空白文案。
- 语言切换采用乐观更新，保存失败仅 toast 提示，不回滚 UI 语言。

## 6) 测试策略

- i18n 初始化单测：验证语言映射与回退逻辑。
- `LanguageSwitcher` 组件测试：点击语言项会调用 `changeLanguage` 并触发保存请求。
- `AppProvider` / `useUser` 集成测试：用户偏好可覆盖浏览器语言。

## 7) 迁移与发布

- 数据库迁移新增 `preferred_language` 列。
- 前端逐步替换核心页面文案为 `t()`。
- 先灰度发布核心路径，后续扩展更多页面与文案。

## 8) 开放问题

- 是否需要在设置页同步增加语言选择入口（首期先不做）。
- 是否需要后端侧缓存更新后主动刷新 session（目前依赖前端与 TTL）。
